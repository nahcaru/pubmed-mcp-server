# 2.6.12 — 2026-05-16

## Added

- **`NCBI_MAX_CONCURRENT` env var** — caps in-flight NCBI requests (default `8`, range 1–16). Decouples concurrency from `NCBI_REQUEST_DELAY_MS` so a slow upstream doesn't block new dispatches behind a single worker. Documented in `.env.example` and the README config table.

## Changed

- **`@cyanheads/mcp-ts-core` ^0.9.0 → ^0.9.1.**
- **`NcbiRequestQueue` rewritten as a token-bucket scheduler** (`src/services/ncbi/request-queue.ts`). Two independent ceilings: `minStartGapMs` enforces NCBI's per-second start-rate cap, `maxConcurrent` caps in-flight requests. `enqueue(task, endpoint, params, signal?)` now accepts an `AbortSignal` and rejects a still-waiting waiter immediately when its signal fires, splicing it out of the queue without dispatching.
- **`NCBI_REQUEST_DELAY_MS` description** refined in `.env.example` and README — names what it actually does ("minimum gap between request starts") instead of the older "delay between requests".
- **Dev deps.** `sanitize-html` ^2.17.3 → ^2.17.4, `@types/node` ^25.7.0 → ^25.8.0.
- **`skills/api-workers/SKILL.md`** 1.3 → 1.4 — documents the `instructions` resolver form `(env: CloudflareBindings) => string` so env-derived orientation text reaches the model inside `initializeApp(env)`, after `injectEnvVars()`.

## Fixed

- **Queue wait now counts toward `totalDeadlineMs`** ([#50](https://github.com/cyanheads/pubmed-mcp-server/issues/50)). `NcbiService.performRequest` and `idConvert` previously constructed `runWithDeadline` *inside* the enqueued task, so during NCBI eutils incidents a queued request could sit for minutes before its deadline started ticking — observed `pubmed_fetch_fulltext` and `pubmed_find_related` durations climbing in lockstep to 10+ minutes. `runWithDeadline` now wraps the enqueue, threading the combined deadline+caller signal into queue wait, retry backoff sleeps, and the HTTP fetch.
- **`pubmed_find_related` similar wedge on high-traffic PMIDs** ([#53](https://github.com/cyanheads/pubmed-mcp-server/issues/53)). NCBI's `cmd=neighbor_score` backend was returning a 200 OK with a `TXCLIENT::readAll EOF` C++ exception trace for popular source PMIDs (1, 12345, 33567185, 37952131), so the retry chain ran for ~60s before surfacing `-32003 ServiceUnavailable`. Switched to `cmd=neighbor` with explicit `linkname=pubmed_pubmed`, which returns the same neighbor list in NCBI's natural relevance order. Output schema drops the `score` field — it was an arbitrary-scale integer, so ordering is preserved without it.
- **`devcheck` outdated-packages parser** — `bun outdated` emits markdown-style rows (`| col1 | ... |`), so package names live at `split('|')[1]`, not `[0]`. The previous parser was matching the empty leading cell as the package name, so the allowlist always missed and every outdated dep surfaced as unexpected. Also strips the trailing `(dev|peer|prod|optional)` workspace-type marker so allowlist entries match the bare name.
